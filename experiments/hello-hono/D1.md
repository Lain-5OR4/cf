# D1 を hello-hono に組み込む手順

Cloudflare D1 (SQLite-on-edge) を `hello-hono` に追加して `/` のページに DB 値を表示するまでの記録。

## 前提

- `bunx wrangler login` 済み(初回のみ。トークンは `~/.wrangler/` に保存される)
- `hello-hono` Worker が動いている (`bun run dev` で起動できる状態)

## 1. D1 データベースを作成

```sh
bunx wrangler d1 create hello-hono-db
```

出力例:

```
✅ Successfully created DB 'hello-hono-db' in region APAC
{
  "d1_databases": [
    {
      "binding": "hello_hono_db",
      "database_name": "hello-hono-db",
      "database_id": "dc2dc864-72f6-4df4-adce-2fabc331e699"
    }
  ]
}
```

`database_id` をメモ。`binding` 名は後で書き換えてOK(本リポでは `DB` に変更)。

## 2. `wrangler.jsonc` にバインディングを追加

```jsonc
"d1_databases": [
  {
    "binding": "DB",
    "database_name": "hello-hono-db",
    "database_id": "dc2dc864-72f6-4df4-adce-2fabc331e699"
  }
]
```

- **`binding`** は Worker コードからアクセスする時の名前(`c.env.DB`)。慣習的に大文字。
- **`database_name`** は人間用の識別子。
- **`database_id`** が実体への参照。これが間違ってると別 DB を見にいく。

## 3. スキーマと seed を `schema.sql` に書く

```sql
DROP TABLE IF EXISTS messages;

CREATE TABLE messages (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  text TEXT NOT NULL,
  created_at TEXT NOT NULL DEFAULT CURRENT_TIMESTAMP
);

INSERT INTO messages (text) VALUES
  ('hello from D1 👋'),
  ('this row was seeded from schema.sql'),
  ('try editing schema.sql and re-running wrangler d1 execute');
```

## 4. ローカルとリモートの D1 にスキーマを適用

```sh
bunx wrangler d1 execute hello-hono-db --file=schema.sql --local --yes
bunx wrangler d1 execute hello-hono-db --file=schema.sql --remote --yes
```

- **`--local`** は `wrangler dev` / vitest が使うローカル SQLite ファイル(`.wrangler/state/v3/d1/...`)
- **`--remote`** は Cloudflare 上の本番 D1
- **両者は完全に独立**。片方だけに適用すると、もう片方で `no such table` になる
- `--yes` でリモート適用時の確認プロンプトをスキップ

## 5. 型を再生成

```sh
bun run cf-typegen      # = wrangler types
```

`worker-configuration.d.ts` が更新され、`Env` に `DB: D1Database` が現れる。
**`wrangler.jsonc` のバインディングを変更したら毎回これを叩く**。

## 6. Hono からクエリする

```ts
import { Hono } from "hono";

type Message = { id: number; text: string; created_at: string };

const app = new Hono<{ Bindings: Env }>();

app.get("/api/messages", async (c) => {
  const { results } = await c.env.DB
    .prepare("SELECT id, text, created_at FROM messages ORDER BY id")
    .all<Message>();
  return c.json(results);
});
```

`prepare(...).all<T>()` がリスト取得、`first<T>()` が単行、`run()` が更新系。
パラメータは `.bind(value1, value2)` で渡すと SQL injection 安全。

## デプロイ

普段通り `git push` するだけ。Workers Builds が:

1. リポジトリを fetch
2. `npx wrangler deploy` を実行
3. `wrangler.jsonc` の `d1_databases` を読んで Worker に DB をバインド

リモート D1 のスキーマは手順4で適用済みなので、デプロイ後すぐにクエリ可能。

## よくある詰まりどころ

| 症状 | 原因 |
|---|---|
| `no such table: messages` (本番) | `--remote` で schema 適用してない |
| `no such table: messages` (ローカル) | `--local` で schema 適用してない |
| `c.env.DB` が `undefined` の型補完 | `bun run cf-typegen` を回し忘れ |
| build log に D1 binding が出ない | `wrangler.jsonc` の JSON 構文ミス(コメント不可・末尾カンマ等) |
| ローカルのデータをリセットしたい | `rm -rf .wrangler/state/v3/d1` してから `--local` で再適用 |

## 便利な D1 コマンド集

```sh
# 任意の SQL を実行
bunx wrangler d1 execute hello-hono-db --remote --command "SELECT * FROM messages"

# DB一覧
bunx wrangler d1 list

# DB削除(取り返しつかないので注意)
bunx wrangler d1 delete hello-hono-db

# バックアップ(エクスポート)
bunx wrangler d1 export hello-hono-db --remote --output=backup.sql
```
